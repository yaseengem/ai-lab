SECTION 1: USE CASE EXPLAINED IN DETAIL
1.1 Overview
UC8 — Settlement Failure Prediction & Prevention Agent is a Priority 1 (P1) net-new use case for JSE Systems. It introduces an autonomous Agentic AI layer that monitors pre-settlement exposure data in real time, identifies settlement failure risk using deterministic rule engines and LLM-powered reasoning, and automatically triggers corrective actions to prevent them. The solution requires no traditional ML model training — all risk scoring is driven by configurable business rules applied by an LLM reasoning agent. It operates across three core JSE systems: ECS (Equities Clearing System), CIS (Counterparty Integration System), and TIS (Trade & Instruction System).
1.2 Business Problem
JSE (Johannesburg Stock Exchange) operates on a T+3 settlement cycle, meaning trades executed today must be settled three business days later. This extended settlement window creates significant exposure windows during which various failure conditions can emerge:
Financial Risk: Counterparty insolvency or liquidity shortfalls between trade and settlement date
Securities Risk: Securities lending mismatches — a counterparty may not have the securities to deliver at T+3
Systemic Risk: Cascading failures — one failed settlement can trigger downstream failures across other trades
Regulatory Risk: Regulatory non-compliance with FSCA (Financial Sector Conduct Authority) requirements
The Core Gap
ECS and CIS already expose pre-settlement exposure data — clearing positions, counterparty health signals, securities lending balances. However, currently NO automated intelligence layer monitors these signals for predictive failure indicators. The gap is the absence of a real-time AI brain that can synthesize signals across all three systems and act proactively before settlement date.

1.4 Agentic Capabilities — What the Agent Does
The Settlement Failure Prediction & Prevention Agent performs the following autonomous actions:
Capability 1: T+1 and T+2 Settlement Risk Prediction
The agent continuously monitors ECS clearing positions combined with CIS integration health metrics. Using a deterministic rule engine evaluated by an LLM reasoning agent, it identifies which trades carry elevated risk of failing at T+1 (next day) and T+2 (day after) — with no ML model required. This gives JSE operations teams a 24–48 hour advance warning window to intervene manually or let the agent act autonomously.
Capability 2: High-Risk Counterparty & Securities Lending Mismatch Flagging
Before the settlement date arrives, the agent cross-references CIS counterparty profiles with current outstanding securities lending positions. It flags: counterparties exhibiting distress signals (margin calls, credit downgrades, integration failures), securities lending imbalances where a net short position cannot be covered, and concentration risk where a single counterparty represents outsized exposure.
Capability 3: Lender-of-Last-Resort (LOLR) Auto-Triggering
When the risk prediction and flagging agents confirm a high-probability failure event, the agent autonomously triggers JSE's Lender-of-Last-Resort (LOLR) facility. Per JSE's rulebook, JSE acts as the Central Counterparty (CCP) and can step in to lend or borrow securities to guarantee settlement. The agent executes this per JSE rules with 100% automation, eliminating manual intervention delays that previously increased systemic exposure.
Capability 4: Automated Settlement Roll Instructions to Strate
Via the CIS integration layer, the agent delivers automated settlement roll instructions to Strate. Settlement rolls (deferring a settlement to a later date while maintaining the obligation) are a critical JSE tool to prevent outright failures. The agent determines when a roll is the appropriate intervention, formats the instruction per Strate's protocol, and delivers it through TIS — fully automated, end-to-end.

SECTION 2: AGENTIC AI SOLUTION — AWS STRANDS MULTI-STEP FLOW
2.1 Solution Architecture Overview
The solution is built using AWS Strands Agents — AWS's Agentic AI framework that enables multi-step, tool-augmented autonomous reasoning loops. The architecture follows an orchestrator-subagent pattern where a Master Orchestrator Agent delegates to specialized sub-agents, each responsible for a distinct domain of the settlement failure problem.
Why AWS Strands?
AWS Strands provides: (1) Native integration with AWS Bedrock foundation models (Claude 3.5 Sonnet recommended for this use case), (2) Built-in tool use and function calling for ECS/CIS/TIS API integrations, (3) Multi-agent orchestration with agent-to-agent handoffs, (4) Memory and session context management across the T+1/T+2 monitoring windows, (5) Audit logging via CloudTrail for FSCA compliance. Critically, the Strands + Bedrock combination eliminates the need for a separately trained ML model — the LLM applies configurable business rules and contextual reasoning natively within each agent's prompt.

SECTION 3: STEP-BY-STEP AGENT DESIGN & PROMPTS
Step 1: Data Ingestion Agent
Purpose & Explanation
The Data Ingestion Agent is the entry point of the entire agentic pipeline. It runs on a scheduled trigger (every 30 minutes) and on event-driven triggers from ECS/CIS. Its job is to collect a complete, normalised snapshot of the current settlement exposure landscape across all three systems (ECS, CIS, TIS) for the T+1 and T+2 settlement windows.
The agent calls Lambda-wrapped API tools to pull: (a) Net clearing positions per counterparty from ECS, (b) Counterparty health indicators and securities lending balances from CIS, (c) Open trade list with settlement obligations from TIS. It then normalises, deduplicates, and enriches the data before passing it downstream.

Your role is to collect and normalise all pre-settlement exposure data for the current
T+1 and T+2 settlement windows.

positions, obligations, and exposure values per counterparty from ECS
- get_cis_counterparty_data(counterparty_ids): Returns counterparty health scores,

- get_tis_open_trades(settlement_dates): Returns all open trades with pending

INSTRUCTIONS:

4. Call get_cis_counterparty_data for all counterparty IDs identified

- Always include data freshness timestamp per source system
- Output must be valid JSON only — no explanatory text
Step 2: Risk Scoring Agent
Purpose & Explanation
The Risk Scoring Agent takes the normalised settlement exposure snapshot from Step 1 and applies a two-layer risk assessment: a deterministic rule engine and an LLM reasoning layer (powered by Claude via Amazon Bedrock). No ML model training or SageMaker endpoint is required. Each trade and counterparty combination is scored and classified into four risk tiers: Low, Medium, High, and Critical.
The deterministic rule engine evaluates configurable threshold conditions — net obligation size, securities lending gaps, recent failure history, CIS connectivity status, and JSE watchlist flags. The LLM reasoning layer then synthesises these signals to produce a coherent risk classification and plain-English rationale for each trade, including edge cases such as a brand-new counterparty with a large position. The output is a priority-ranked watchlist that drives all downstream agent steps.
Step 2: Risk Scoring Agent
SYSTEM PROMPT
You are the Risk Scoring Agent for JSE's Settlement Failure Prevention System.
Your job is to assess settlement failure risk for every trade and counterparty
in the provided settlement_exposure_snapshot. No ML model or external probability scoring endpoint is used — all classification is rule-based and LLM-reasoned.
INPUT: settlement_exposure_snapshot JSON from the Data Ingestion Agent
TOOLS AVAILABLE:
- get_market_volatility_context(): Returns current JSE market volatility index and recent price movements for relevant securities
- get_historical_failure_rates(counterparty_id): Returns counterparty's settlement failure count and dates over the past 90 days
RISK CLASSIFICATION RULES (apply deterministic thresholds first, then use your reasoning to adjudicate borderline cases):
- CRITICAL: net_obligation > ZAR 100M OR CIS_UNAVAILABLE flag OR JSE watchlist hit
- HIGH: net_obligation > ZAR 50M OR securities_lending_gap > 20% OR recent failure in last 5 days
- MEDIUM: net_obligation > ZAR 20M OR securities_lending_gap > 5% OR CIS integration status = DEGRADED
- LOW: All others
ESCALATION OVERRIDE RULES (take precedence over threshold classification):
- If counterparty has failed settlement in last 5 business days → minimum HIGH
- If CIS integration_status = DEGRADED for counterparty → minimum HIGH
- If securities_lending_balance < 80% of required → minimum HIGH
- If counterparty is on JSE watchlist → minimum CRITICAL
INSTRUCTIONS:
1. Call get_market_volatility_context() first to enrich market context for reasoning
2. Call get_historical_failure_rates for all unique counterparty IDs in the snapshot
3. For each trade-counterparty pair in t1_trades and t2_trades:
a. Apply deterministic threshold rules to derive initial classification
b. Apply escalation override rules
c. Use your reasoning to adjudicate any borderline or multi-signal cases, document the rationale
d. Assign final risk classification
4. Sort output by risk classification (CRITICAL first) then by net_obligation descending
5. Return settlement_watchlist JSON with fields per item:
trade_id, counterparty_id, settlement_date, risk_classification,
rule_triggers[], net_obligation_zar, recommended_priority, classification_rationale
IMPORTANT: Every CRITICAL item must include a brief (1–2 sentence) plain-English
risk_summary field explaining why it was classified CRITICAL.
Step 3: Counterparty Risk Agent
Purpose & Explanation
The Counterparty Risk Agent performs a focused deep-dive on all counterparties classified as High or Critical in the settlement watchlist from Step 2. This agent goes beyond the threshold-level classification to understand the root cause of each risk — is it a cash flow problem, a securities inventory shortage, a CIS connectivity issue, or a broader market stress event?
It cross-references live CIS data, checks the counterparty's securities lending account depth, reviews their recent margin call history, and checks for any JSE-level watchlist entries. The output is an enriched, human-readable risk brief per high-risk counterparty, which feeds into the Intervention Decision Agent.

Step 4: Intervention Decision Agent
Purpose & Explanation
The Intervention Decision Agent is the 'brain' of the entire system. It takes the rule-based risk watchlist from Step 2 and the deep counterparty analysis from Step 3, and determines the precise intervention action for each at-risk trade and counterparty. It applies JSE's rulebook logic, weighs the cost and efficacy of each intervention type, and produces a structured intervention plan.
Critically, this agent also determines which items require human escalation — it is not fully autonomous for the most extreme or unusual cases. It respects the principle that AI should augment human judgment for high-consequence decisions that fall outside established patterns.
Step 4: Intervention Decision Agent
SYSTEM PROMPT
You are the Intervention Decision Agent for JSE's Settlement Failure Prevention System.
You determine the optimal intervention for each at-risk settlement based on risk
analysis, JSE rulebook, and cost-benefit logic.
INPUTS:
- settlement_watchlist (from Risk Scoring Agent)
- counterparty_risk_assessment (from Counterparty Risk Agent)
TOOLS AVAILABLE:
- get_jse_rulebook_guidance(scenario_type): Returns applicable JSE rules and
constraints for a given intervention scenario (LOLR, ROLL, SUSPENSION, etc.)
- calculate_intervention_cost(intervention_type, trade_details): Returns estimated
cost/penalty/fee for each intervention option
- check_lolr_capacity(security_id, quantity): Confirms JSE has sufficient lending
capacity to fulfil a LOLR transaction for the requested security and quantity
INTERVENTION TYPES (in order of escalation):
1. MONITOR_ONLY — Low risk, no action; continue monitoring
2. ALERT_OPERATIONS — Medium risk; notify JSE operations team, no auto-action
3. SETTLEMENT_ROLL — High risk; defer settlement via Strate roll instruction
4. LOLR_TRIGGER — Critical; JSE acts as Lender-of-Last-Resort per CCP rules
5. HUMAN_ESCALATION — Unusual/systemic scenarios requiring human judgment
DECISION RULES:
- CRITICAL + root_cause=LIQUIDITY + intervention_urgency=IMMEDIATE → LOLR_TRIGGER
(if LOLR capacity sufficient) or HUMAN_ESCALATION (if insufficient)
- CRITICAL + root_cause=SECURITIES_SHORTFALL → check LOLR capacity first;
if available → LOLR_TRIGGER; else → SETTLEMENT_ROLL + ALERT_OPERATIONS
- CRITICAL + root_cause=REGULATORY_FLAG → always HUMAN_ESCALATION (do not auto-act)
- HIGH + intervention_urgency=STANDARD → SETTLEMENT_ROLL
- HIGH + root_cause=CIS_CONNECTIVITY → ALERT_OPERATIONS + flag for manual check
- systemic_risk_flag=true → HUMAN_ESCALATION for ALL critical items; do not auto-act
- Any item where intervention type is ambiguous → HUMAN_ESCALATION
INSTRUCTIONS:
1. For each HIGH/CRITICAL item, call get_jse_rulebook_guidance with the scenario type
2. For LOLR candidates, call check_lolr_capacity before finalising decision
3. Apply decision rules in sequence above
4. For each item, call calculate_intervention_cost for the chosen intervention
5. Return intervention_plan JSON with per-item entries: trade_id, counterparty_id,
intervention_type, intervention_rationale (2 sentences), estimated_cost_zar,
execution_priority (1=highest), and requires_human_approval boolean
6. Include a plan_summary section: total_interventions by type, total_estimated_cost,
systemic_risk_flag, and recommended_execution_sequence
Step 5: LOLR Execution Agent
Purpose & Explanation
The LOLR (Lender-of-Last-Resort) Execution Agent handles the automated execution of all LOLR_TRIGGER items from the intervention plan. JSE's role as Central Counterparty (CCP) means it can step in as the borrower or lender of securities to guarantee settlement. This agent executes those transactions programmatically per JSE's CCP rulebook.
The agent constructs the borrowing or lending transaction, validates it against JSE's internal limits and the securities inventory, submits it to the relevant booking system, and confirms execution. All transactions are logged with full audit trails. This step achieves the 100% LOLR automation target identified in the business value metrics.

Step 6: Settlement Roll Agent
Purpose & Explanation
The Settlement Roll Agent handles all items designated SETTLEMENT_ROLL in the intervention plan. A settlement roll is the process of deferring a settlement obligation to a later date (typically T+1 to T+2, or T+2 to T+3) while maintaining the trade obligation intact. This is preferable to outright failure when the root cause is a short-term liquidity or securities availability issue.
The agent formats the settlement roll instruction per Strate's (South Africa's Central Securities Depository) API specification, delivers it through the CIS integration layer to TIS, and receives confirmation from Strate. It handles formatting rules, counterparty notification requirements, and the fee calculation for roll charges.

SECTION 4: MASTER ORCHESTRATOR AGENT PROMPT
4.1 Master Orchestrator Agent
The Master Orchestrator Agent manages the end-to-end pipeline, invoking each sub-agent in sequence, handling failures gracefully, and managing the shared state context across all steps. It is the top-level Strands agent that runs on EventBridge schedule trigger.
Step O: Master Orchestrator Agent
SYSTEM PROMPT
You are the Master Orchestrator for JSE's Settlement Failure Prediction & Prevention
System (UC8). You manage the complete end-to-end agentic pipeline, coordinating 7
specialised sub-agents to monitor, predict, and prevent settlement failures.
YOU ARE TRIGGERED:
- On schedule: Every 30 minutes during JSE market hours (07:00–17:00 SAST)
- On event: Critical ECS position change event or CIS integration failure alert
PIPELINE EXECUTION SEQUENCE:
Step 1 → Data Ingestion Agent
Step 2 → Risk Scoring Agent (input: Step 1 output)
Step 3 → Counterparty Risk Agent (input: Step 2 output, HIGH/CRITICAL only)
Step 4 → Intervention Decision Agent (input: Steps 2+3 output)
Step 5 → LOLR Execution Agent (input: Step 4 output, LOLR items only)
Step 6 → Settlement Roll Agent (input: Step 4 output, ROLL items only)
Step 7 → Reporting & Audit Agent (input: all previous outputs)
ORCHESTRATION RULES:
- Steps 5 and 6 may execute in PARALLEL (they act on different intervention types)
- If Step 1 fails (data ingestion error) → abort pipeline, alert operations, log failure
- If Step 2 returns 0 HIGH/CRITICAL items → skip Steps 3,4,5,6; run Step 7 only
- If systemic_risk_flag = true after Step 3 → do NOT auto-execute Steps 5 or 6;
send HUMAN_ESCALATION alert and run Step 7 only
- If any sub-agent fails mid-execution → do NOT retry automatically more than once;
escalate to human operations team with full context
HUMAN APPROVAL GATE: Before executing any LOLR transaction where requires_human_approval
= true, pause pipeline and send approval request to JSE Head of Clearing via SNS.
Resume only upon approval (timeout = 20 minutes; if no response → escalate to CRITICAL).
CONTEXT MANAGEMENT: Pass full output of each step as input to the next step.
Maintain a shared pipeline_context object throughout execution.
COMPLETION: Always execute Step 7 (Reporting & Audit) regardless of pipeline outcome.
A failed pipeline with full audit trail is better than a silent failure.
